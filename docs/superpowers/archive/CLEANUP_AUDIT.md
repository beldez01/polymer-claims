#### 1. `grammar/src/polymer_grammar/description_length.py:154-167`

- **What:** The whole guard that makes deprecating a load-bearing pattern RAISE the corpus code is `total += fill * _n_structural_slots(c)` with `fill = _GENERIC_FILL_BITS` for generic-repointed claims (lines 165-166). But `_n_structural_slots` (lines 91-96) counts ONLY QuantityLeaf/PropositionLeaf. A claim built from categorical-only or existence-only leaves has 0 structural slots, so the inflated fill multiplies to 0 — no penalty at all. Worse, after a pattern-DEPRECATE the repointed claims keep the SAME pattern-selector frequency (their old pattern's count == the new __generic__ count when it was the only deprecated pattern), and their categorical term selectors are unchanged, so `_corpus_code_length` is byte-for-byte unchanged while `_schema_cost` drops by `_PATTERN_BITS` (+log* delta). Net mdl_delta is negative -> `clears_mdl_bar` True and `classify` returns 'consolidation' for what should be a rejected load-bearing deprecation. The existing test `test_loadbearing_deprecate_is_rejected` only passes because `_dup_corpus`/`_qleaf` give every claim a QuantityLeaf (1 slot, +28 bits each). The intent stated in the module/transport docstrings ('deprecating a load-bearing specific pattern RAISES the corpus code') is violated for slotless claims.
- **Proposed:** Make the deprecation penalty independent of the structural-slot count: either price each generic-repointed claim with a flat per-claim `_GENERIC_FILL_BITS` floor (e.g. `fill * max(_n_structural_slots(c), 1)` for generic-pointed claims) or add a separate flat per-generic-claim term, so a slotless load-bearing pattern still incurs a strictly positive penalty larger than the schema saving. Mirror the same floor for the term-DEPRECATE path if categorical-only claims can also be slotless. FLAG: this changes mdl_delta/classification numbers for corpora containing slotless claims (currently none in golden tests), so confirm no MDL value is golden-captured before applying.
